1.HTTP 和 HTTPS 的区别是什么？HTTPS 是如何实现安全传输的？

HTTP和HTTPS是两个不同的网络协议，其中HTTP表示超文本传输协议，而HTTPS表示超文本传输安全协议。它们之间的主要区别在于安全性。

HTTP是一种明文传输协议，因此数据在传输过程中容易被黑客截取和窃取。HTTPS在HTTP基础上添加了安全层，使用SSL/TLS协议来加密数据，从而保证了传输数据的安全性。

HTTPS实现安全传输的过程如下：

  1. 客户端向服务器发送一个连接请求。

  2. 服务器将自己的证书（包含公钥）发送给客户端。

  3. 客户端对服务器的证书进行验证，包括证书的有效性、真实性等。

  4. 如果证书验证通过，客户端会生成一个对称密钥，并使用服务器的公钥进行加密，然后发送给服务器。

  5. 服务器使用自己的私钥对加密后的对称密钥进行解密，并保存在服务器端。

  6. 服务器使用对称密钥加密数据，并发送给客户端。

  7. 客户端使用对称密钥解密数据，完成传输。

通过这个过程，HTTPS保证了传输过程中的数据加密和安全性，有效防止了黑客的攻击和窃取。

2.TCP 和 UDP 的区别是什么？它们各自适用于什么场景？

TCP（传输控制协议）和UDP（用户数据报协议）是互联网传输层中常用的两个协议，它们有以下几点区别：

  1. 连接方式：TCP是面向连接的协议，需要先建立连接，然后再进行数据传输；UDP是无连接的协议，发送数据之前不需要先建立连接。

  2. 可靠性：TCP是一种可靠的协议，它通过数据校验、确认、重传等机制来确保数据的可靠传输；UDP是不可靠的，不能保证数据一定能够被接收方接收到。

  3. 传输效率：TCP在保证可靠性的前提下，传输效率相对较低；UDP则以高效为目标，传输效率相对较高。

  4. 适用场景：TCP适用于对数据可靠性要求较高的场景，如文件传输、邮件传输等；UDP适用于实时性要求较高的场景，如视频直播、语音通话等。

综上所述，TCP和UDP各有优缺点，应根据实际场景进行选择和使用。

3.OSI 七层网络模型和 TCP/IP 四层网络模型有什么区别？它们各自有哪些层次？

OSI 七层网络模型和 TCP/IP 四层网络模型是两种不同的网络协议分层体系。

OSI 七层网络模型包括以下七层：

  1. 物理层（Physical Layer）：处理比特流传输，将数字信号转化成物理信号。

  2. 数据链路层（Data Link Layer）：提供数据帧传输和识别功能。

  3. 网络层（Network Layer）：提供数据包传输、路由和寻址。

  4. 传输层（Transport Layer）：提供端到端的可靠传输，例如 TCP。

  5. 会话层（Session Layer）：提供会话管理和同步。

  6. 表示层（Presentation Layer）：数据格式转换、数据加密和解密等。

  7. 应用层（Application Layer）：提供应用程序间的通信服务，例如 HTTP、FTP 等。

TCP/IP 四层网络模型包括以下四层：

  1. 网络接口层（Network Interface Layer）：负责物理设备和网络之间的接口。

  2. 网络层（Internet Layer）：负责 IP 协议、地址解析和路由。

  3. 传输层（Transport Layer）：提供端到端的可靠传输，例如 TCP。

  4. 应用层（Application Layer）：提供应用程序间的通信服务，例如 HTTP、FTP 等。

相比于 TCP/IP 四层网络模型，OSI 七层网络模型更加细致，其中的会话层、表示层和应用层都有更加明确的定义。但是，在实际应用中，TCP/IP 四层网络模型更加普遍，因为它更加简单和实用。

4.什么是 RESTful API？它的设计原则是什么？

RESTful API 是一种基于 REST（Representational State Transfer，表现层状态转化）架构风格的 API。它是一种轻量级的、可扩展的、面向资源的 Web 服务设计风格，可以在分布式网络环境中构建和访问 Web 资源。

RESTful API 的设计原则包括：

  1. 状态转移：客户端和服务器之间传递的是资源的状态，客户端在请求中指定资源的状态转移。

  2. 资源：每个资源都有一个唯一的标识符（URI），客户端通过 URI 来操作资源。

  3. 统一接口：对资源的操作具有一致的接口设计，包括标准的 HTTP 方法和状态码。

  4. 超文本驱动：客户端可以通过服务器返回的数据来发现资源和操作。

  5. 自描述消息：服务器返回的消息包含足够的信息，使得客户端可以理解如何处理响应。

  6. 无状态：服务器不存储客户端的状态信息，客户端每次请求都需要提供全部的信息。

RESTful API 的设计原则能够让客户端和服务器之间的通信更加简单、灵活、可扩展和可维护。

5.DNS 的作用是什么？DNS 解析的过程是什么？

DNS（Domain Name System）是互联网上进行域名和 IP 地址解析的服务，它将域名映射成对应的 IP 地址。

DNS 解析的过程通常包括以下步骤：

  1. 浏览器从 URL 中获取域名。
  2. 浏览器向本地 DNS 服务器（通常由网络服务提供商提供）发出请求，询问该域名对应的 IP 地址。
  3. 如果本地 DNS 服务器中有该域名的解析记录，则返回该 IP 地址；否则本地 DNS 服务器向根 DNS 服务器发出请求。
  4. 根 DNS 服务器返回一个 TLD（顶级域名）的 DNS 服务器地址。
  5. 本地 DNS 服务器向 TLD DNS 服务器发出请求，请求返回对应域名的 DNS 服务器地址。
  6. 本地 DNS 服务器向获得的 DNS 服务器发出请求，请求返回对应域名的 IP 地址。
  7. 如果上述步骤都无法找到对应的 IP 地址，则返回一个错误。

DNS 解析的过程可以在命令行中使用 nslookup 命令进行测试。例如，nslookup www.google.com 可以查看 www.google.com 对应的 IP 地址。

6.什么是跨域？如何解决跨域问题？有哪些跨域攻击方式？

跨域是指在浏览器同源策略的限制下，访问不同域名、端口或协议的资源时产生的安全问题。同源策略是浏览器的一项基本安全机制，它限制了浏览器在加载来自不同源的资源时的行为，防止恶意网站窃取用户的数据或发起攻击。

跨域问题的解决方法有多种，其中比较常用的包括：

  1. JSONP（JSON with Padding）：利用<script>标签的跨域特性，将要传递的数据包裹在一个JavaScript函数调用中，以函数返回值的形式传递数据。

  2. CORS（Cross-Origin Resource Sharing）：服务器在响应请求时设置响应头，允许跨域请求访问资源。

  3. 代理：在同源策略允许的情况下，前端发送请求到同源服务器，由服务器代为请求跨域资源，并将结果返回给前端。

跨域攻击方式包括：

  1. CSRF（Cross-Site Request Forgery）跨站请求伪造：攻击者盗用用户的身份，以用户的名义发送恶意请求，从而在未经授权的情况下执行某些操作，如发布文章、发消息等。

  2. XSS（Cross-Site Scripting）跨站脚本攻击：攻击者将恶意脚本注入到正常的网页中，当用户浏览该网页时，恶意脚本就会执行，从而进行窃取用户信息、发送恶意请求等操作。

7.什么是 CDN？它的作用是什么？

CDN是Content Delivery Network（内容分发网络）的缩写，它是一种分布式的网络架构，通过将静态资源（如图片、视频、JavaScript、CSS等）缓存在离用户最近的节点上，实现快速高效的内容传输。

CDN的作用主要有两个方面：

1.提高网站访问速度和用户体验：CDN可以缓存静态资源并部署在全球各地的服务器上，这样当用户访问网站时，可以从距离最近的服务器获取资源，从而提高访问速度和用户体验。

2.减轻原始服务器负载压力：使用CDN可以将静态资源的请求分散到各个节点，减轻原始服务器的负载压力，提高服务器的稳定性和可用性。

CDN的实现原理是将静态资源缓存到多个服务器上，当用户请求资源时，会根据用户的位置和服务器的负载情况，选择距离最近、负载最小的服务器提供服务。这样可以减少网络延迟和带宽使用，提高资源的访问速度和可用性。

CDN的常用实现方式包括：DNS负载均衡、反向代理、内容复制、分发协调等。

8.什么是 Websocket？它与 HTTP 的区别是什么？它的应用场景有哪些？

Websocket 是一种基于 TCP 的协议，它允许在单个 TCP 连接上进行全双工通信，实现客户端和服务器之间的实时数据传输。

与 HTTP 相比，Websocket 的主要区别有：

  1. Websocket 是一个全双工通信协议，而 HTTP 是一个半双工通信协议。

  2. Websocket 是一个持久连接协议，客户端和服务器可以保持连接状态，并随时发送和接收数据。而 HTTP 是一种非持久连接协议，每次请求都需要建立一个新的连接。

Websocket 的应用场景包括：

  1. 实时通信：Websocket 可以实现客户端和服务器之间的实时双向通信，可以用于实时聊天、在线游戏等场景。

  2. 实时数据更新：Websocket 可以在服务器端数据发生变化时主动向客户端推送数据，可以用于股票行情、新闻资讯等场景。

  3. 视频直播：Websocket 可以用于实现视频直播场景，客户端和服务器之间可以实时传输视频流数据。

解决跨域问题时，Websocket 的跨域方案与 HTTP 的方案略有不同。一般来说，Websocket 的跨域问题可以通过在服务器端配置跨域访问控制（CORS）来解决，也可以通过代理服务器来解决。

拓展：全双工通信是指通信的双方都可以同时发送和接收数据的通信方式，即通信双方可以同时发送和接收数据，不需要等待对方的回复才能发送下一条信息。这种通信方式可以实现实时交互，不需要等待延迟，同时还能够避免数据的丢失。在网络通信、电话通信、对讲机等通信领域都有广泛的应用。

9.网络安全方面的问题，例如 XSS、CSRF、DDoS 等攻击方式，以及如何预防这些攻击。

网络安全是前端工程师必须掌握的一项技能，以下是关于 XSS、CSRF、DDoS 攻击方式以及预防措施的简单介绍：

  1. XSS（跨站脚本攻击）：攻击者在网站中注入恶意脚本，当用户访问该网站时，恶意脚本会被执行，从而达到窃取用户信息、操作用户账户等目的。

预防措施：

  • 对用户输入数据进行过滤和验证，避免用户输入恶意脚本。
  • 将用户输入的数据进行转义，例如将 < 转义为 &lt;，避免恶意脚本被执行。
  • 在 HTTP 响应头中设置 Content-Security-Policy，限制页面中可以加载的资源类型和来源。
  • 对于使用了第三方库和插件的网站，及时更新版本，避免已知的漏洞被攻击利用。
  1. CSRF（跨站请求伪造攻击）：攻击者通过欺骗用户在已登录网站上执行恶意操作，例如转账、修改密码等。

预防措施：

  • 在表单提交时使用 CSRF Token，确保请求来源是合法的。
  • 避免在 GET 请求中进行敏感操作。
  • 在 HTTP 响应头中设置 SameSite 属性为 Strict 或 Lax，限制第三方网站获取 Cookie。
  • 避免使用 Cookie 存储敏感信息，使用 HTTP-Only 属性限制 Cookie 只能通过 HTTP 访问。
  1. DDoS（分布式拒绝服务攻击）：攻击者通过控制大量的计算机向目标服务器发起请求，使得服务器瘫痪。

预防措施：

  • 增加服务器带宽和硬件资源，提高抗攻击能力。
  • 使用防火墙和入侵检测系统（IDS）等安全设备，过滤恶意请求。
  • 使用 CDN 加速，分散请求流量，减轻服务器负载。
  • 及时更新服务器软件和操作系统，避免已知漏洞被攻击利用。

以上是简单介绍，实际应用中需要根据具体情况选择合适的预防措施。同时，前端工程师也需要不断学习和关注网络安全方面的新技术和漏洞。

10.tcp的三次握手和四次挥手，用简单形象的话语描述

三次握手：
两个人要开始通话，需要先打个招呼确认对方是存在的。第一次握手就是A发个消息给B，告诉B我要开始通信了。第二次握手是B发消息给A，告诉A我已经收到你的消息了，我也准备好了。第三次握手是A再次发消息给B，告诉B我也收到你的消息了，我们可以开始通信了。

四次挥手：
两个人通信完毕，需要结束通信。第一次挥手是A发个消息给B，告诉B我通信结束了。第二次挥手是B发消息给A，告诉A我收到你的消息了。第三次挥手是B发消息给A，告诉A我也准备好结束通信了。第四次挥手是A发消息给B，告诉B我也收到你的消息了，我们结束通信了。

专业版：

TCP 的三次握手和四次挥手是建立和终止网络连接时的重要步骤。它们的具体过程如下：

  1. 三次握手：

第一次握手：客户端向服务器发送 SYN 请求，表示客户端请求建立连接。

第二次握手：服务器收到客户端的 SYN 请求后，发送 SYN+ACK 请求，表示服务器接收到了请求，并且准备好建立连接。

第三次握手：客户端收到服务器的 SYN+ACK 请求后，向服务器发送 ACK 请求，表示客户端已经准备好建立连接。

  1. 四次挥手：

第一次挥手：客户端向服务器发送 FIN 请求，表示客户端请求终止连接。

第二次挥手：服务器收到客户端的 FIN 请求后，向客户端发送 ACK 请求，表示服务器已经收到了请求。

第三次挥手：服务器向客户端发送 FIN 请求，表示服务器请求终止连接。

第四次挥手：客户端收到服务器的 FIN 请求后，向服务器发送 ACK 请求，表示客户端已经收到了请求。

三次握手的过程中，客户端和服务器都必须确认对方的身份，并同意建立连接。四次挥手的过程中，双方都必须发送 FIN 请求，表示终止连接。

这样的设计可以确保连接的可靠性和稳定性，防止数据的丢失和冗余传输。同时，也可以有效避免网络拥塞和资源浪费的问题，提高网络传输效率和可靠性。

11.Http 状态码 301 和 302 的应用场景分别是什么

HTTP 状态码 301 和 302 都是重定向状态码，用于指示客户端需要重定向到另一个资源。

HTTP 状态码 301 表示永久重定向。它告诉客户端请求的资源已经被永久移动到了一个新的 URL，且这个 URL 是该资源的新位置。对于搜索引擎来说，它会将旧 URL 的权重转移到新 URL 上，建议使用 301 状态码。

HTTP 状态码 302 表示临时重定向。它告诉客户端请求的资源暂时被移动到了另一个 URL，但未来可能还会返回到原来的 URL 上。对于搜索引擎来说，它不会将旧 URL 的权重转移到新 URL 上，建议使用 301 状态码。

在实际应用中，301 状态码常用于网站 URL 重构、网站迁移等操作，而 302 状态码则常用于网站的临时跳转，例如一个正在维护的页面暂时跳转到一个备份页面上。

12.cookie 和 token 都存放在 header 中，为什么不会劫持 token？

Cookies 和 Token 都可以被存放在 HTTP 请求头中，但是它们的作用和机制不同。

Cookie 是一种服务器存储在客户端的小型文本文件，用于跟踪用户活动和存储用户信息。当客户端向服务器发送请求时，会自动将之前存储在客户端的 Cookie 信息添加到请求头中一起发送给服务器，这样服务器就可以通过解析 Cookie 获取用户信息。然而，由于 Cookie 存在浏览器中，因此可能被黑客通过 XSS 攻击或者其他方式进行窃取或篡改。

Token 是一种用于身份验证和授权的字符串，它由服务器生成，并在客户端存储。Token 的生成和解析过程是在服务器端完成的，客户端只需要将 Token 添加到请求头中发送给服务器。服务器接收到请求后会解析 Token，从而验证用户身份和权限。相比于 Cookie，Token 的优势在于可以存储在客户端任意位置（例如 LocalStorage、SessionStorage 等），从而避免了 CSRF 攻击和 XSS 攻击等安全问题。同时，使用 Token 的方式也可以避免跨域问题。

因此，尽管 Cookie 和 Token 都可以存放在请求头中，但是由于 Token 的生成和解析过程是在服务器端完成的，因此即使黑客拿到了 Token，也无法篡改其内容。而 Cookie 的存在可能会存在安全隐患，因此在使用 Cookie 时需要格外注意安全问题。

13.GET 和 POST 的区别

GET和POST是HTTP请求方法，它们之间的区别如下：

  1. GET请求用于从服务器获取资源，而POST请求用于向服务器提交数据。
  2. GET请求将参数附加到URL的查询字符串中，而POST请求将参数放在请求体中。
  3. GET请求是幂等的，也就是说多次请求相同的URL得到的响应是一样的；而POST请求不是幂等的，因为每个请求都可能会对服务器上的资源进行修改。
  4. GET请求通常用于获取数据，而POST请求通常用于提交数据，例如表单数据。
  5. GET请求参数有长度限制，因为URL长度有限制；而POST请求则没有长度限制。

总的来说，GET请求适合获取资源，而POST请求适合提交数据。在实际应用中，我们应该根据具体的业务需求选择合适的请求方法。

14 在浏览器地址栏输入地址，并按下回车键后，发生了哪些事情？

完整的浏览器地址栏输入地址，按下回车键后，发生的步骤如下：

  1. 浏览器会检查输入的地址是否合法，如果不合法则会给出错误提示。
  2. 浏览器会检查当前是否有缓存可用，如果有且未过期，则直接从缓存中读取页面资源并显示。
  3. 如果没有缓存或缓存过期，则浏览器会进行 DNS 查询，解析出对应的 IP 地址。
  4. 浏览器向服务器发起 TCP 连接，并发送 HTTP 请求。
  5. 服务器接收到请求后，解析 HTTP 请求头部，根据请求头部信息进行请求处理，生成相应的 HTTP 响应。
  6. 服务器将生成的 HTTP 响应返回给浏览器。
  7. 浏览器接收到 HTTP 响应，解析响应头部信息，根据响应头部的状态码进行处理，如 200 表示成功，302 表示重定向，404 表示资源未找到等。
  8. 如果响应头部中包含缓存相关信息，则浏览器会将资源缓存起来，以便下次使用。
  9. 浏览器解析响应体中的 HTML 文档，并开始渲染页面，加载文档中的其他资源，如 CSS、JavaScript、图片等。
  10. 页面加载完成后，浏览器会触发 load 事件，并执行相应的 JavaScript 代码。

15.cookie/sessionStorage/localStorage 的区别

cookie、sessionStorage和localStorage都是浏览器提供的用于存储数据的技术，但它们之间有一些区别，主要有以下几点：

  1. 存储容量：

cookie的存储容量最大为4KB，而sessionStorage和localStorage的存储容量一般为5MB或更多。

  1. 生命周期：

cookie的生命周期一般为浏览器关闭后失效，但可以设置cookie的过期时间，使其在指定的时间过后失效。sessionStorage的生命周期为在同一个会话中有效，当用户关闭浏览器窗口时会被清除。localStorage的生命周期一般没有过期时间，除非用户手动清除。

  1. 存储位置：

cookie数据存储在客户端浏览器中，并在每次请求时都会发送给服务器，因此不安全。而sessionStorage和localStorage数据也存储在客户端浏览器中，但不会随每次请求一起发送给服务器。

  1. 存储数据类型：

cookie只能存储字符串类型的数据，而sessionStorage和localStorage可以存储任何JavaScript支持的数据类型。

  1. 作用域：

cookie的作用域可以是整个域名下的所有页面，也可以是单个页面。sessionStorage和localStorage的作用域都是当前源（即协议+域名+端口号）下的所有页面。

总之，三者的用途不同，cookie一般用于客户端和服务器之间的数据交互，而sessionStorage和localStorage一般用于客户端存储用户相关的数据。

16.http缓存

HTTP缓存指的是浏览器或客户端在向服务器发起请求时，会首先检查本地是否有缓存，如果有缓存且未过期，则直接使用本地缓存，否则再向服务器发起请求获取最新资源。

HTTP缓存可以分为两种：强制缓存和协商缓存。

强制缓存是通过设置HTTP响应头信息中的Cache-Control和Expires字段来控制的。Cache-Control和Expires可以指定资源缓存的过期时间，过期时间内再次请求相同的资源时直接从缓存中获取，不再向服务器发起请求。

协商缓存是通过比较本地缓存和服务端资源的Last-Modified/If-Modified-Since、ETag/If-None-Match等字段的值来控制的。当第一次请求资源时，服务器会将资源的Last-Modified和ETag值一同返回给浏览器，当再次请求资源时，浏览器会带上If-Modified-Since和If-None-Match等条件字段与服务端进行比较，如果服务端资源没有发生变化，则返回304状态码，浏览器直接使用本地缓存。如果服务端资源发生变化，则返回200状态码，并将新的资源内容和新的Last-Modified/ETag值返回给浏览器，浏览器更新本地缓存。

使用HTTP缓存可以大幅度提高网站的性能和用户体验，减少了不必要的网络请求，减轻了服务器压力。但需要注意的是，如果缓存时间过长或者服务端资源更新频繁而没有及时更新缓存，则可能导致本地缓存数据与实际数据不一致，影响网站正常运行，因此需要根据具体的业务场景来合理设置缓存时间。

浏览器缓存策略是浏览器为了提高网站性能和减少网络传输而采取的一些策略。主要有以下几种：

  1. 强制缓存：浏览器直接从缓存中读取资源，并不发起请求到服务器。强制缓存过期时间由响应头中的Expires和Cache-Control字段控制，当过期时间内再次访问该资源时，仍然从缓存中读取，不请求服务器。

  2. 协商缓存：在强制缓存失效的情况下，浏览器通过向服务器发送请求来判断该资源是否被修改过。如果未被修改，服务器返回304状态码，告诉浏览器直接从缓存中获取资源。协商缓存通过比较请求头和响应头中的Last-Modified和ETag字段来实现。

  3. 离线缓存：在支持HTML5的浏览器中，可以使用AppCache实现离线缓存。它通过将网站资源缓存在本地，使得用户在没有网络连接的情况下也能够访问网站。

  4. 预加载：在支持HTML5的浏览器中，可以使用link标签的rel="preload"属性来进行预加载。预加载可以在页面加载完毕后异步加载资源，提高网站的性能。

  5. 其他：浏览器还有一些其他的缓存策略，如TCP/IP连接复用、DNS预解析等，可以进一步提高网站性能和减少网络传输。
